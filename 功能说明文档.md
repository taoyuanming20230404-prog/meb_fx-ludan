# 分销后台表单批量录单功能说明

## 1. 文档目的

本文档用于说明 `fx_ludan.py` 的功能范围、运行方式、输入输出、业务流程与当前限制，便于业务人员、维护人员和后续开发人员快速理解程序。

## 2. 程序概述

本程序是一个基于 Python + Selenium 的自动化录单工具，用于将 Excel 中的客户数据批量录入分销后台或分享表单页面。

程序当前提供两类录单能力：

1. 流程一：先在分销后台执行重单校验，再进行录单。
2. 流程二：直接在分享表单页录单，不做重单校验。

其中，当前 `main()` 中实际写死为 `flow_choice = 1`，也就是默认只走“流程一”。

## 3. 技术组成

- Python
- pandas：读取 Excel 数据
- openpyxl：Excel 输出支持
- selenium：浏览器自动化操作
- rapidfuzz：项目关键词模糊匹配
- tkinter：弹窗选择 Excel 文件
- pyautogui / pywin32：辅助浏览器窗口置前，非必需

## 4. 主要业务目标

程序围绕“批量录单”完成以下任务：

1. 读取 Excel 客户数据。
2. 打开 Chrome 并进入分销后台页面。
3. 等待用户登录后台。
4. 按行处理客户数据。
5. 根据“项目”字段匹配标准项目关键词。
6. 在页面中填写手机号、城市、项目、说明、微信等信息。
7. 提交表单并记录处理状态。
8. 输出录单反馈 Excel、详细日志、重单记录。

## 5. 输入说明

### 5.1 Excel 输入文件

程序通过文件选择框选择一个 `.xlsx` 文件作为输入。

首行必须为表头，代码中实际使用到的列有：

- `号码`：必填；为空时该行会被跳过。
- `项目`：用于匹配后台项目关键词，并写入客户说明。
- `城市`：用于选择城市。
- `微信`：非必填；有值时尝试填写。

### 5.2 项目匹配词库

程序会读取以下配置文件：

- `std_keywords.txt`：标准项目关键词列表，每行一个
- `synonyms.json`：项目同义词映射

读取优先级：

1. 先读程序运行目录下的外部文件。
2. 如果是打包环境，再尝试读取打包资源中的同名文件。
3. 如果标准词库读取失败，则回退到默认项目关键词。

## 6. 项目匹配规则

项目字段不会直接原样用于选择后台项目，而是会先经过匹配逻辑，输出一个标准项目关键词。

匹配顺序如下：

1. 精确包含匹配：如果原始项目文本中直接包含某个标准关键词，则取最长命中的标准词。
2. 同义词匹配：如果命中 `synonyms.json` 中的同义词键，则映射为对应标准词。
3. 模糊匹配：如果安装了 `rapidfuzz`，则使用 `partial_ratio` 和 `token_sort_ratio` 做模糊匹配。
4. 默认兜底：如果都未命中，则返回默认项目关键词。

程序还提供 `test_project_match.py`，用于离线测试项目匹配效果，并输出匹配结果表。

## 7. 页面与流程说明

### 7.1 启动阶段

程序启动后会：

1. 初始化日志。
2. 启动 Chrome。
3. 打开重单查询页面 `https://hhr.meb.com/manage/#/customer/recommend`。
4. 等待用户在浏览器中完成登录。
5. 弹窗选择 Excel 文件。
6. 开始批量处理。

如果未来启用流程二，程序还会同时打开分享录单页：

- `https://hhr.meb.pw/manage/mobile/#/share?id=1356139172909420544`

### 7.2 流程一：重单校验 + 后台录单

这是当前版本默认执行的流程。

处理逻辑如下：

1. 切换或打开“重单查询”标签页。
2. 逐行读取 Excel 数据。
3. 若手机号为空，标记为“跳过”。
4. 输入手机号并等待页面反馈是否为重单。
5. 如果检测为重单：
   - 不继续提交录单；
   - 记录到重单文本文件；
   - 当前行状态记为“重单”。
6. 如果不是重单：
   - 填写微信（如有）；
   - 填写客户称呼；
   - 按项目匹配规则选择项目；
   - 填写客户说明；
   - 选择城市；
   - 点击提交；
   - 检测成功或失败提示；
   - 记录结果状态。

### 7.3 流程二：分享页直接录单

流程二的设计目标是直接在分享页填写表单，不执行重单校验。

字段包括：

- 称呼
- 手机号
- 城市
- 项目
- 说明
- 微信

提交后程序也会尝试识别成功或失败提示，并输出反馈结果。

注意：当前主流程未开放用户选择流程，实际运行时默认固定为流程一。

## 8. 输出结果

### 8.1 录单反馈 Excel

程序会在原 Excel 所在目录输出一份反馈文件：

- 文件名格式：`录单反馈+YYYYMMDD.xlsx`

该文件是在原始表基础上新增一列状态，用于标记每一行的处理结果。

状态值包括：

- `已录单`
- `重单`
- `跳过`
- `失败`

### 8.2 详细日志文件

程序会在原 Excel 所在目录输出详细日志文件：

- 文件名格式：`日志YYYYMMDDHHMMSS.txt`

日志会记录每个手机号对应的处理结果和失败原因，便于排查。

### 8.3 重单记录文件

流程一中如果出现重单，程序会生成文本文件：

- 文件名格式：`重单记录_YYYY-MM-DD.txt`

同时会在当月目录下复制保存一份，例如：

- `2026-04\重单记录_2026-04-09.txt`

若同一天多次运行，会自动加 `_2`、`_3` 等后缀，避免覆盖。

### 8.4 月份目录

程序会按当前年月创建文件夹，例如：

- `2026-04`

用于保存当月的重单记录副本。

## 9. 状态判定规则

程序主要依赖页面提示来判断提交结果。

成功提示检测范围包括：

- Element UI 成功消息
- Notification 成功提示
- 含有“成功”“提交成功”等文本的节点
- 常见 Toast 提示

失败提示检测范围包括：

- Element UI 错误消息
- 表单错误提示
- 含有“失败”“错误”“无法”等文本的节点

若短时间内未检测到明确提示，程序会保留当前状态并记录告警日志。

## 10. 用户操作方式

实际运行时，用户操作步骤如下：

1. 启动程序。
2. 等待 Chrome 自动打开。
3. 在分销后台页面手动登录。
4. 回到控制台按回车继续。
5. 在弹窗中选择 Excel 文件。
6. 等待程序自动逐行处理。
7. 到 Excel 目录查看反馈文件和日志。

## 11. 异常处理机制

程序已实现的主要异常处理包括：

- 浏览器连接断开识别
- 页面元素查找失败记录
- Excel 读取失败记录
- 输出文件保存失败记录
- 字段缺失时使用默认值或跳过
- 浏览器页面异常后尝试重新回到目标页面

当识别到浏览器连接已经断开时，程序会中止执行，避免无意义重试。

## 12. 当前实现限制

基于当前代码，存在以下实际限制：

1. 主流程被固定为流程一，用户无法在运行时自由切换流程。
2. 登录逻辑中存在写死的账号密码占位值，自动登录功能不可直接用于生产。
3. 录单页面元素定位大量依赖固定 CSS/XPath，页面改版后容易失效。
4. 反馈 Excel 的命名按日期生成，同一天重复运行会覆盖前一次结果。
5. 流程二的提交逻辑对微信字段缩进敏感，当前实现存在“无微信时可能不提交”的风险。

## 13. 适用场景

适用于以下场景：

- 运营人员根据收集表批量导入客户线索
- 需要先做手机号重单筛查再录单
- 需要对项目字段进行标准化匹配
- 需要保留每批录单的处理结果与失败原因

## 14. 维护建议

建议后续维护重点关注以下方面：

1. 将流程选择恢复为可配置项。
2. 将账号密码改为外部配置或人工登录，不写入源码。
3. 为页面定位增加更稳健的兜底策略。
4. 为反馈文件增加防覆盖命名规则。
5. 为流程二补充自动化测试或修正提交逻辑。

## 15. 代码入口与关键函数

关键入口及职责如下：

- `main()`：程序总入口
- `launch_new_browser()`：启动浏览器并打开页面
- `process_jinshuju_excel_new()`：读取 Excel 并调度流程
- `run_flow1_duplicate_check_and_entry()`：流程一主逻辑
- `run_flow2_entry_only()`：流程二主逻辑
- `get_project_keyword()`：项目关键词匹配
- `_save_feedback_excel()`：输出录单反馈表
- `_create_detail_logger()`：生成详细日志
- `_detect_submit_success()`：检测页面提交结果

## 16. 总结

该程序已经具备“Excel 读取、项目标准化匹配、后台自动录单、重单记录、结果回写、日志留痕”等完整能力，适合在固定业务页面下执行批量录单任务。

从实现成熟度看，流程一已经是当前主用路径；流程二和自动登录能力仍需要进一步修正和加固，才能稳定作为正式能力交付。
